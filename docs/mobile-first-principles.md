# Mobile-First Product Principles

These principles guide the future daily execution experience without defining screens, routes, navigation, components, or design tokens.

1. **Daily execution must work well on a phone.** The initial product remains a responsive web application; mobile-first does not imply a native app.
2. **Speed and clarity outrank density.** The daily surface should emphasize immediate action rather than dense analytics or the entire planning hierarchy.
3. **Common actions need minimal navigation.** Reviewing priorities, completing work, deferring supported items, and opening the check-in should be direct.
4. **Controls must be touch-friendly.** Targets, spacing, feedback, and error recovery should work reliably during one-handed use.
5. **Brief visits are a primary use case.** The daily view should remain useful when the user has only seconds to check or update the day.
6. **Priorities need clear distinction.** Today's most important work should be visually distinct from lower-priority or contextual items.
7. **Daily input must stay lightweight.** The system should not require a long daily form. Daily check-ins are skippable unless later evidence supports stronger requirements.
8. **Planning can be denser elsewhere.** Season planning and historical analysis may use richer desktop layouts later without burdening the daily surface.
9. **Accessibility remains foundational.** Touch optimization must preserve keyboard access, semantic structure, focus visibility, readable contrast, and responsive behavior.
10. **Views do not own records.** The daily surface composes information and invokes domain actions; it is not a competing source of truth.
11. **Uncertainty remains visible.** The daily view's name, route, final navigation position, landing-page role, and visual design are undecided.
